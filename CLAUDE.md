### 需求分级与 PR 判断

每次接收到新需求时，按以下标准判断是否需要开 PR：

| 需求类型 | 是否开PR | 示例 |
|---------|---------|------|
| **新功能开发** | ✅ 必须 | 添加拖拽定位、新增按钮类型 |
| **架构重构** | ✅ 必须 | 模块拆分、API重构 |
| **UI/交互优化** | ✅ 必须 | 交互逻辑改变、新增设置项 |
| **Bug修复** | ❌ 可直接提交 | 修复崩溃、显示异常、逻辑错误 |
| **文档更新** | ❌ 可直接提交 | README、开发文档维护 |
| **代码微调** | ❌ 可直接提交 | 变量命名、注释补充、格式整理 |

### 开发流程

1. **需求分析** → 判断是否需要开 PR
2. **若需开 PR**：
   - 创建功能分支（命名如 `feature/hud-drag-position`）
   - 创建需求方案文档（存放于 `开发文档/` 目录）
   - 开发、测试、调试
   - 完成后创建 PR，等待用户确认合并
3. **跟进合并** — PR 创建后主动询问用户是否合并入 main
4. **若不需开 PR** → 直接在 main 分支完成并提交

### 合并确认机制

- PR 创建完成后，必须询问用户：「PR 已创建，是否合并入 main 分支？」
- 用户确认后才执行合并操作
- 合并后及时更新相关开发文档

## 模块化开发文档管理

每次开发新功能或新需求时，按以下流程处理：

1. **分析模块归属** — 判断该功能属于哪个现有模块
2. **文档操作**：
   - 有对应模块 → 更新该模块的开发文档
   - 无对应模块 → 新建开发文档（命名清晰，如 `HUD交互模块.md`）
3. **持续维护** — 后续围绕该模块的任何优化、调整、Bug修复，都要同步更新对应的开发文档

**目的**：确保开发文档与代码实现保持同步，方便追溯和查阅。

## 发布打包流程

### 自动触发规则

当用户回复以下关键词时，**自动询问是否需要打包发布**：
- `ok`、`好的`、`可以`、`没问题`
- `解决了`、`修好了`、`正常了`
- `完成`、`测试通过`、`效果不错`

### 打包流程

用户确认打包 → 执行以下步骤：

1. **确认版本号一致**（两处）：
   - `src/main.py` 中的 `setApplicationVersion("0.1.0")`
   - 如不一致，询问用户是否需要更新版本号

2. **执行打包命令**：
   - Windows：`python build.py`
   - macOS：暂不支持（需另行配置）

3. **复制到发布目录**：
   - 输出位置：`release/v{版本号}/`
   - 文件名：`MMY_SvnGo_v{版本号}_{时间戳}.exe`

4. **生成 README.txt**（包含版本号、时间戳、文件列表）

### 发布后提示

告知用户：
- 发布目录位置：`release/v{版本号}/`
- 可用于分发的文件列表
